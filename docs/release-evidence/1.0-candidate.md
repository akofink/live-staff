## Evidence Labels

- **Automated** means a repeatable local or CI check with browser or audio interfaces replaced where the test requires control.
- **Emulated viewport** means desktop headless Chromium rendered at specified CSS dimensions; it is not physical mobile-browser evidence.
- **Desktop browser inspection** means production was inspected in headless Chromium on the named desktop host; it is not an attended Chrome, Safari, or screen-reader session.
- **Physical device** means an identified device and operating system were used directly by a person.

- Offline continued use after initial load without a service worker, Cache Storage, or requests from listening and preference changes.

These tests establish application behavior under controlled browser objects.
They do not establish operating-system permission UI, backgrounding, screen lock, interruption, route switching, physical microphone cleanup, mobile power use, or detector accuracy on a live source.

## Production Inspection

Headless Chromium loaded the production custom domain and requested only the first-party document, CSS, JavaScript, notation chunk, and a missing first-party favicon during the inspected initial-load and denied-permission flow.
The only console error was the first-party favicon `404`.
No fetch request was observed after changing application state, and the production privacy verifier reported no telemetry or third-party executable request.
The verifier checks known telemetry patterns and third-party executable resources during initial load; it does not prove the absence of arbitrary first-party transmission during active microphone capture.

The accessibility tree exposed:

- A `main` landmark, named Live Staff region, named listening-control region, and named detected-pitch region.
- A start button described by a polite atomic status region.
- An accessible grand-staff label and hidden text for empty history.
- A named setup disclosure, instrument combobox, input-filter group and controls, calibration button, diagnostics disclosure, and privacy heading.

Activating Start listening moved the status through `Requesting microphone access...` to `Microphone access was denied. Allow access and try again.` and retained focus on the start button.
This verifies DOM accessibility semantics and focus behavior for the denied path, not that a screen reader announced either message.

Keyboard Tab reached the setup summary, which displayed a 3 CSS px solid focus outline, and Enter expanded it.
The DevTools bridge enforced a 500 CSS px minimum when asked for 320 px, so this production session is not 320 px evidence.

## Known Blockers And Limitations

- Issue [#77](https://github.com/akofink/live-staff/issues/77) reports a baseline of 3 matching fixtures out of 10 in-range fixtures and 20 octave errors among 31 emitted estimates; this run did not retain an independent fixture artifact for those counts.
- The fixture threshold prevents regression but does not support a general or real-device accuracy statement.
- No current physical browser configuration has passing evidence in this record.
- No attended desktop Safari or Chrome session has exercised a real microphone.
- No VoiceOver, TalkBack, NVDA, JAWS, or other screen reader was used.
- No sustained active capture measured memory growth, battery drain, dropped updates, or thermal behavior.
- No background, screen-lock, phone-call, OS interruption, permission-revocation, wired-route, Bluetooth-route, or external-microphone-loss scenario was performed on real hardware.
- No live voice, generated acoustic tone, piano, or transposing instrument was evaluated in a production browser during this run.

## Operator Checklists

Record the date, production commit, device model, OS version, browser version, input route, duration, result, and limitation for every checklist run.
Attach screenshots only after checking that they do not expose notifications, account data, device identifiers, or other private information.
Use the [developer-only attended evidence harness](../attended-release-evidence.md) to persist explicit human attestations locally and export versioned, validated JSON and Markdown reports.
The harness does not replace any physical test and never infers a pass.

### iPhone Or iPad Safari

- Confirm issue #77 has merged and record the deployed commit before evaluating accuracy.
- Load production online, then verify the idle staff, primary control, detected-pitch text, setup disclosure, and privacy copy at the device's narrow portrait width and in landscape.
- Start only from the button, allow microphone access in Safari UI, play voice or a generated acoustic tone, and record the displayed and expected concert pitch without generalizing beyond tested notes.
- Select B-flat trumpet and verify the written note is a whole step above the same concert source; confirm history and accessible text update together.
- Add and edit a filter, compare bypass, calibrate in quiet, and enable then disable the monitor.
- Stop and verify the browser microphone indicator clears.
- Repeat after background and resume, screen lock and unlock, an OS audio interruption, permission revocation, wired route change, Bluetooth route change, and external-input removal when hardware permits.
- Run at least 30 minutes with filters and monitor enabled; record responsiveness, memory information if Safari Web Inspector exposes it, battery change, thermal observation, and any dropped or frozen display.
- Inspect Safari Web Inspector network traffic from before Start through Stop; fail the check for any audio, detection, preference, profile, analytics, or third-party executable request.

### Android Chrome

- Confirm issue #77 has merged and record the deployed commit before evaluating accuracy.
- Record the phone model, Android version, Chrome version, physical display mode, and whether the CSS viewport is at least 320 px.
- Repeat the core, transposition, history, calibration, filters, bypass, monitor, stop, and privacy-network flows from the iOS checklist.
- Repeat after app switching, screen lock, OS audio interruption, site-permission revocation, wired route change, Bluetooth route change, and external-input removal when supported.
- Run at least 30 minutes with filters and monitor enabled; record Chrome task memory when available, battery change, thermal observation, responsiveness, and display continuity.
- Inspect remote DevTools network traffic from before Start through Stop and fail on prohibited transmission.

### Screen Reader And Keyboard

- Use a named screen reader and browser combination on physical desktop or mobile hardware.
- Navigate from page start without a pointer and confirm logical focus order, visible focus, native control names, units, values, disabled state, and disclosure state.
- Confirm idle, requesting-permission, listening, silence or uncertain input, detected note, permission denial, device loss, and recovery messages are announced once and remain understandable.
- Confirm the visual staff is not read as raw SVG content and that current note, written or concert context, staff, and recent history are available as text.
- Edit all filter control types, toggle bypass, perform calibration, and enable and disable diagnostics without a pointer.
- Repeat at 200 percent zoom or the platform's large-text setting and confirm no control or status is clipped or obscured.

### Release Decision

- Do not close issue #71 while any required physical-device, screen-reader, sustained-performance, or privacy-network row remains missing.
- Do not claim supported iOS, Android, Safari, screen-reader, thermal, route-switch, or real-source accuracy behavior from automated Chromium evidence.
- Do not cut 1.0 until issue #77 is merged, the affected deployed commit is retested, and the release review accepts the resulting limitations.
